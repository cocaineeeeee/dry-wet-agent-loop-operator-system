## SEAM 2 — time-series observable schema (**the biggest / novel seam**)
**Shared files:** `expos/adapters/wet/sim_reader.py` (or a new neutral observation carrier), the
observation-ingest path, `expos/kernel/objects.py` observation shape (READ-ONLY confirm neutrality).
- The observation for M26 is a **derived dynamic-phenotype SUMMARY of a trace**, not a single
  reading. M26's convention (kernel-neutral, mirrors the M24 value/secondary split):
  `RawResult.value` = the load-bearing dynamic scalar (cassette steady-state / toggle bistable
  separation); `RawResult.secondary` = `{steady_state, response_amplitude, switching_time,
  separation}`. **The raw trace stays in the domain layer / content-store; the kernel only ever
  sees numbers.** So *no kernel object change is strictly required* for v0.1 — confirm this and
  bless the convention, OR add a first-class `trace_ref` (content-store URI for the full trace)
  to the observation provenance if you want the trace auditable. M26 recommends: content-store
  the trace (domain layer), keep kernel on value+secondary (zero kernel change), same as M24.
- Derivation is centralised in `circuit_dynamics.derive_phenotype` (used by BOTH legs) so dry
  and wet summarise a trace identically — the single seam point if the summary schema evolves.

## SEAM 3 — dynamic truth faces registration (high / flipped / flat)
**Shared file:** `expos/adapters/wet/sim_reader.py` (`TRUTH_PROFILES`) and/or a dynamic reader.
- M26 defines the three DYNAMIC faces LOCALLY in `timeseries_reader.py::DYNAMIC_TRUTH_PROFILES`
  (`dynamic_high`=0.85, `dynamic_flipped`=0.20, `dynamic_flat`=null) obeying the **only-mu-differs**
  K-D law (proven in tests 5). They are NOT yet in the shared `sim_reader.TRUTH_PROFILES`, so the
  yaml's `acceptance_faces` are `status: declared`, not `landed`.
- Decision for B: either (a) add the three faces to the shared `TRUTH_PROFILES` and teach
  `sim_reader` to emit a TIME-SERIES per reading for `genetic_circuit` (the reader currently emits
  one scalar per well), or (b) keep `timeseries_reader` as a **separate dynamic reader process**
  registered alongside `sim_reader` and driven out-of-band by mcl. M26 recommends (b): the dynamic
  trace generation is genuinely different from the scalar plate reader; a sibling reader keeps
  `sim_reader` byte-unchanged (regression anchor) while reusing the same only-mu-differs law.

## SEAM 4 — mcl wet-leg wiring (drive the time-series reader out-of-band)

## SEAM 5 — verify gate placement between propose and dry (curriculum gate)
**Shared file:** `expos/mcl.py` (proposal→dry hand-off)
- Insert a call to `domains.genetic_circuit.verify.verify(graph)` (or a neutral `verify_hook`
  the provider exposes) **after propose, before the dry leg**, so an illegal topology is rejected
  cheaply and never reaches simulation. M26 already enforces the gate INSIDE the dry adapter
  (`CircuitTopologyAdapter.enforce_verify`, test 6b) as a belt-and-braces fallback, but the
  architecturally correct placement is the propose→dry seam (docs/bio_refs/02 §4). The verify
  verdict is **dry evidence only — it never certifies a claim** (red line, BIOLOGY_PROGRAM_2026 §4).
  Suggested neutral shape: provider optionally exposes `verify_proposal(payload) -> {ok, failed_level,
  highest_passed}`; mcl treats a non-ok verdict as a dropped/abstained candidate, logged, no wet cost.

## SEAM 6 — replicate aleatoric vs epistemic (intrinsic noise ≠ independent evidence)
**Shared files:** `expos/qc/*` replicate-collapse decision (READ path already exists for M24).
- docs/bio_refs/02 §3: the stochastic-proxy's **intrinsic noise is aleatoric / within-replicate**
  (a technical axis) and must NOT be counted as independent biological evidence; independence comes
  from the **environmental/context** axis (epistemic). The yaml declares `replicate_kind: biological`
  (each well an independent reaction). If a future protocol re-reads ONE stochastic trajectory across
  wells, those are TECHNICAL replicates and must be collapsed by `expos.qc.replicate_collapse` —
  same `cfg.replicate_kind` threading M24 already requested. No new mechanism; confirm reuse.
