## SEAM 2 — model-competition backend dispatch + baseline-gate wiring (the M27 core)
**Shared file:** `expos/mcl.py` (whatever orchestrates candidate scoring / promotion).
- M27's `BioModelBackend` contract (`expos/adapters/models/virtual_cell.py`) is the delegation of
  B's existing `agent_backend` / model-scorer pattern to the perturbation modality. Each backend
  is a **proposer/scorer only** (emits `ResponsePrediction`, NEVER writes a claim). B's seam:
  drive the three registered backends (mean/linear/knn) per round, collect their predictions as
  **dry evidence / proposals**, and route them through the competition layer
  (`domains/perturbation/competition.py`: `score_backend` + `baseline_gate`). The gate's
  `GateVerdict.negative_claim` (a rejected, `kind="baseline_gate_negative"` claim) must enter the
  **existing** claim ledger as a first-class negative claim (charter §4 — not a leaderboard row).
- Requirement: an **expensive proposer that does not clear the gate must NOT influence acquisition**
  (only baselines + admitted backends vote — `run_v01` shows the voter set). B decides where in the
  promotion path the "admitted models only steer selection" rule sits. The competition math is
  domain-local + tested; B only needs the dispatch seam + the negative-claim ledger entry.
- The five decision faces (`performance/calibration/biological_fidelity/ood_abstention/
  experiment_cost`) are computed domain-locally (`FaceScores`); no kernel change — they read only
  predictions + reference deltas. `experiment_cost` is a placeholder proxy; if you want real
  compute-cost provenance, expose a per-backend cost hook (optional, not v0.1-blocking).

## SEAM 3 — response-DISTRIBUTION observable + causal-claim certification path
**Shared files:** the observation-ingest path; `expos/kernel/objects.py` observation shape
(READ-ONLY confirm neutrality); `claims.py` (existing lifecycle, no ledger change expected).
- The M27 observable is a **per-axis response vector** (delta-from-control), and a claim is
  per-(perturbation, axis): "knockout X moves axis A up/down". M27's kernel-neutral convention
  (mirrors M24/M26 value/secondary split): `RawResult.value` = the load-bearing response magnitude
  (L2 of the delta), `RawResult.secondary` = a bounded summary (e.g. top-effect axis id + signed
  size). **The full per-axis delta vector + the model proposals stay in the domain layer /
  content-store; the kernel only ever sees numbers.** Confirm this is sufficient for v0.1 (M27
  recommends it — zero kernel change), OR add a first-class `response_ref` (content-store URI for
  the full delta vector) to observation provenance if you want the vector auditable.
- Causal certification is domain-local (`domains/perturbation/causal.py`): only a **trusted
  observation** may set a claim's status; a model prediction is recorded ONLY in the claim's
  `proposals` list (non-certifying dry evidence). B's seam: route a *trusted* perturbation
  observation (a wet/sim reader observation entering mcl) into `certify_causal_claims` /
  `update_claim_with_observation` so certification runs on the REAL trusted-observation path
  rather than on replay (v0.1 currently exercises the lifecycle with retrospective replay, HONESTLY
  labeled non-wet in every claim's `evidence`).

## SEAM 4 — retrospective replay ingest boundary (dual-role guard)
**Shared files:** the ingest/provider path that could surface a dataset as an observation.
- `datasets/replay/*` datasets carry `DatasetProvenance(is_wet_observation=False,
  validation_level='retrospective', role='benchmark_calibration')`, and the constructor HARD-REFUSES
  `is_wet_observation=True`. Requirement: the ingest path must **never** promote a replay dataset to
  a trusted this-run observation — replay is benchmark/calibration only (charter §4). The provenance
  fingerprint (which folds `scope`, e.g. the "SYNTHETIC / cancer-line / embedding-regime" context
  boundary) must ride into `config_fingerprint` so a scope/regime change flips run identity.
- Model/dataset/weight versions into provenance: each backend's `fingerprint()` = `name@version#
  sha256:<weight-hash>`, each dataset's `fingerprint()` folds provenance+data bytes. B: fold both
  into the run fingerprint chain (same mechanism as `provider_fingerprint`).
