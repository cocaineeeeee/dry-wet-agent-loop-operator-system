# M28 biology_discovery — seam list (v0.1 COMPLETE, domain-local)

Team M28 "Understand". v0.1 delivered as a **domain-local autonomous-discovery loop over
the REAL kernel claim ledger**. Unlike the sibling milestones (which carry a domain-local
string-status claim), M28 drives `expos.kernel.claims` directly: agents emit **evidence**
and `ClaimDelta` **proposals**, and the ONLY ledger mutation in the whole organ is the
`apply_claim_deltas` call inside `ledger_bridge.certify_round`. **The ledger is the moat.**

## What ships (all in `agents/biology_discovery/` + `hypotheses/` + `analysis_backends/`)

- **Four deepened agents** (`agents/biology_discovery/agents.py`):
  - `HypothesisAgent` — literature-grounded `DiscoveryContext` → ≥2 COMPETING
    machine-readable hypotheses, each with an `AssayPredicate` (assayable claim) + an
    `AnalysisPlan`, prioritised by a **Robin-adapted** pairwise ranker.
  - `AnalysisAgent` — hypothesis + assay dataset → `EvidenceObservation` (a statistic).
    Produces **evidence only**; the `analysis_backends`/`hypotheses` packages do not import
    `expos.kernel.claims` at all (a guard test asserts this) → structurally incapable of
    certifying.
  - `ContradictionAgent` — flags mutually-exclusive hypotheses and routes competing
    evidence through the ledger, where a contradiction is resolved as a **supersede** under
    the kernel's strength-monotonicity gate (weak may not retract strong).
  - `ReplicationAgent` — enforces the **independent-biological-replicate** bar (technical
    replicates never masquerade) and selects the follow-up from the CHANGED knowledge.
- **`hypotheses/`** — typed hypothesis objects + a deterministic proposer (+ LLM seam) +
  `ranking.py` (a ~30-line pure-Python Bradley-Terry MM estimator ADAPTED from Robin's
  `choix.ilsr_pairwise` usage — kept as **proposer prioritisation only, never evidence**;
  no `choix` dependency transplanted).
- **`analysis_backends/`** — `EvidenceObservation`/`AssayDataset` objects, the
  `AnalysisBackend` protocol + evidence-strength band rule, a deterministic SMD backend,
  and a labelled synthetic (`simulation`) dataset generator.
- **`ledger_bridge.py`** — THE single crossing point: registers the decision_fn
  `m28_discovery_verdict` v1 (via the public `register_decision_fn` decorator — the same
  extension seam `qc/certification_stats.py` and `scripts/claim_compiler.py` use), builds
  `ClaimDelta`s with full W3C-PROV five-tuple provenance, and lands them via
  `apply_claim_deltas`.
- **`run_v01.py`** — the runnable e2e (`python -m agents.biology_discovery`): context →
  ≥2 competing hypotheses (literature prior favours the WRONG direction) → trusted evidence
  → **up SUPPORTED / down REJECTED** (a wrong moderate lead SUPERSEDED at very_strong) →
  weak/technical counter-evidence **denied** by the strength gate → untrusted →
  **INSUFFICIENT** (non-mutating) → knowledge change flips the follow-up
  `run_assay → probe_new_axis`. Machine-readable JSON report.
- **Test**: `tests/test_m28_discovery_v01.py` (8 tests, green) — ≥2 competing; leaf packages
  can't import kernel; analysis/build_delta mutate nothing; supersede + weak-cannot-retract;
  supported+rejected+insufficient separation; determinism; technical-replicate rejection.

Honesty: trusted observations here are **retrospective/simulation stand-ins**
(`is_wet_observation=False`), labelled as such throughout. NOT wet-lab validated.

## Kernel untouched

`expos/kernel/claims.py` is imported READ-ONLY (called, never edited). The registered
decision_fn is **biology-agnostic** — it sees only `effect / se / z / predicted_sign`; the
kernel never learns it is biology. No shared central file (mcl/domain/kernel/planner/qc/
adapters) was edited.

## Seams for the integration owner (B)

1. **trusted-observation source** — `EvidenceObservation.trusted` is currently set by the
   run harness (`dataclasses.replace(obs, trusted=True)`) as the simulation-boundary
   stand-in. Real seam: an adapter from a certified round `Observation` entering `mcl`
   (wet or sim-reader, adjudicated by QC) → `EvidenceObservation(trusted=True)`. Only that
   should feed a decisive band. Analysis of untrusted data already collapses to
   INSUFFICIENT — the red line is enforced agent-side; B keeps the certified stream the
   sole source of `trusted=True`.
2. **event stream / exactly-once** — `certify_round` lands deltas but does NOT yet emit the
   `claim_decision` event. Seam: wire its outcomes through `claims.emit_claim_decision`
   (→ `RunStore.append_decision_face_event`, resume-idempotent exactly-once, dedup key
   `(round_id, claim_id)`, content fp = provenance fp). The provenance five-tuple is already
   fully populated, so this is a wiring step, not a schema change.
3. **mcl orchestration** — the loop is domain-local (`run_v01` owns its `Ledger()`). Seam:
   thread the run's ledger + compiled-knowledge fingerprint through `mcl` so the M28 agents
   run as a proposer/analyzer stage of `run_mcl_loop`; the hypotheses should ride the shared
   `_domain_hypotheses` / `HypothesisObject` substrate (a converter + the stable
   `make_claim_id` scheme are provided).
4. **LLM proposer/analysis backends** — `HypothesisProposerBackend` and `AnalysisBackend`
   are dependency-injected protocols (default deterministic/template, reproducible). Seam:
   inject the real `expos.agent` backend as an optional LLM proposer/analyzer, keeping the
   template default. Even with an LLM, output stays proposals/evidence — the LLM **never**
   writes a narrative claim (moat).
5. **replicate_kind from domain** — `ReplicationAgent(min_reps)` + the biological/technical
   distinction should read the domain's `replicate_kind` / independence assumption so the
   replicate count matches the certification's independence model (align with M24-B).
6. **knowledge fingerprint alignment** — `ledger_bridge.knowledge_fingerprint` hashes the
   ledger's derived `effective_statuses`. Seam: align it with
   `kernel.knowledge.compile_knowledge`'s fingerprint so `consumed_knowledge_fingerprint`
   on M28 deltas matches the run's compiled-knowledge fingerprint exactly.
7. **decision_fn governance sign-off** — confirm `m28_discovery_verdict` v1 in the shared
   `DECISION_FN_REGISTRY` is acceptable as-is (it is generic + versioned + source-
   fingerprinted like the existing entries), or route M28 verdicts through an existing
   registered fn if B prefers a single certification family.
