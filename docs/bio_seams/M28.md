# M28 biology_discovery — seam list (v0.1 COMPLETE, domain-local)

Team M28 "Understand". v0.1 delivered as a **domain-local autonomous-discovery loop over
the REAL kernel claim ledger**. Unlike the sibling milestones (which carry a domain-local
string-status claim), M28 drives `expos.kernel.claims` directly: agents emit **evidence**
and `ClaimDelta` **proposals**, and the ONLY ledger mutation in the whole organ is the
`apply_claim_deltas` call inside `ledger_bridge.certify_round`. **The ledger is the moat.**

## What ships (all in `agents/biology_discovery/` + `hypotheses/` + `analysis_backends/`)

- **Four deepened agents** (`agents/biology_discovery/agents.py`):
  - `HypothesisAgent` — literature-grounded `DiscoveryContext` → ≥2 COMPETING
    machine-readable hypotheses, each with an `AssayPredicate` (assayable claim) + an
    `AnalysisPlan`, prioritised by a **Robin-adapted** pairwise ranker.
  - `AnalysisAgent` — hypothesis + assay dataset → `EvidenceObservation` (a statistic).
    Produces **evidence only**; the `analysis_backends`/`hypotheses` packages do not import
    `expos.kernel.claims` at all (a guard test asserts this) → structurally incapable of
    certifying.
  - `ContradictionAgent` — flags mutually-exclusive hypotheses and routes competing
    evidence through the ledger, where a contradiction is resolved as a **supersede** under
    the kernel's strength-monotonicity gate (weak may not retract strong).
  - `ReplicationAgent` — enforces the **independent-biological-replicate** bar (technical
    replicates never masquerade) and selects the follow-up from the CHANGED knowledge.
- **`hypotheses/`** — typed hypothesis objects + a deterministic proposer (+ LLM seam) +
  `ranking.py` (a ~30-line pure-Python Bradley-Terry MM estimator ADAPTED from Robin's
  `choix.ilsr_pairwise` usage — kept as **proposer prioritisation only, never evidence**;
  no `choix` dependency transplanted).
- **`analysis_backends/`** — `EvidenceObservation`/`AssayDataset` objects, the
  `AnalysisBackend` protocol + evidence-strength band rule, a deterministic SMD backend,
  and a labelled synthetic (`simulation`) dataset generator.
- **`ledger_bridge.py`** — THE single crossing point: registers the decision_fn
  `m28_discovery_verdict` v1 (via the public `register_decision_fn` decorator — the same
  extension seam `qc/certification_stats.py` and `scripts/claim_compiler.py` use), builds
  `ClaimDelta`s with full W3C-PROV five-tuple provenance, and lands them via
  `apply_claim_deltas`.
- **`run_v01.py`** — the runnable e2e (`python -m agents.biology_discovery`): context →
  ≥2 competing hypotheses (literature prior favours the WRONG direction) → trusted evidence
  → **up SUPPORTED / down REJECTED** (a wrong moderate lead SUPERSEDED at very_strong) →
  weak/technical counter-evidence **denied** by the strength gate → untrusted →
  **INSUFFICIENT** (non-mutating) → knowledge change flips the follow-up
  `run_assay → probe_new_axis`. Machine-readable JSON report.
- **Test**: `tests/test_m28_discovery_v01.py` (8 tests, green) — ≥2 competing; leaf packages
  can't import kernel; analysis/build_delta mutate nothing; supersede + weak-cannot-retract;
  supported+rejected+insufficient separation; determinism; technical-replicate rejection.

Honesty: trusted observations here are **retrospective/simulation stand-ins**
(`is_wet_observation=False`), labelled as such throughout. NOT wet-lab validated.

## B's connection point — discovery run entry + verdict→ClaimDelta bridge (M28 handoff)

Answering red_to_blue/156 (*"discovery 域/run 存在后，trusted AnalysisVerdict 走现成
`_certify_round`→ClaimDelta（kernel 门唯一 mutator）"*). Both halves are now in place, and
the connection is **zero-edit** on your side (mcl / planner / kernel untouched):

**1. Discovery run entry (exists, runs).** `python -m agents.biology_discovery`
(`run_v01.run()`) is the domain-local runnable: ≥2 competing hypotheses → **up SUPPORTED /
down REJECTED / untrusted INSUFFICIENT**, separated through the REAL `expos.kernel.claims`.
M28 is a **certification loop, not a candidate-screening loop**, so its mcl integration is
the **seventh planner element (`CertificationPolicy`)**, NOT a `DomainProvider` + domain
yaml (that seam belongs to the screening organs M25/M27). No discovery yaml/provider is
needed or shipped — the bridge below is the entry.

**2. verdict→ClaimDelta bridge — `DiscoveryCertification` (the drop-in `CertificationPolicy`).**
`agents/biology_discovery/certification.py::DiscoveryCertification` conforms **structurally**
to `expos.planner.certification.CertificationPolicy` (a test asserts
`isinstance(cert, CertificationPolicy)`), so you inject it exactly like
`RegisteredFnCertification`:

```python
from agents.biology_discovery import DiscoveryCertification, DiscoveryVerdict

cert = DiscoveryCertification(
    [DiscoveryVerdict(hypothesis, trusted_analysis_obs), ...],   # this round's verdicts
    run_fingerprint=run_fp,
)
# your existing mcl._certify_round already does exactly this:
#   deltas, new_state = cert.decide(adjudicated_observations, ledger,
#                                   cross_round_state, round_id, knowledge_fingerprint)
#   new_ledger, report = apply_claim_deltas(ledger, deltas)   # kernel gate = sole mutator
#   for outcome in report.outcomes: emit_claim_decision(...)
```

- `decide(...)` returns `(list[ClaimDelta], cross_round_state)` — the exact
  `CertificationPolicy.decide` signature `_certify_round` calls. It **only builds proposals**
  (via `ledger_bridge.build_round_deltas` → `build_delta`); the single `apply_claim_deltas`
  mutation stays inside your `_certify_round`. The moat is intact: no agent/LLM touches the
  ledger; the kernel gate is the sole mutator.
- The verdicts' decision_fn `m28_discovery_verdict` v1 is registered in the shared
  `DECISION_FN_REGISTRY` on import (biology-agnostic), so your governance gate accepts these
  deltas like any other. One `DiscoveryVerdict` → one delta; a competing pair → two deltas →
  the gate certifies one SUPPORTED, REJECTS the other (supersede under the strength gate).
- **Knowledge-fingerprint threading (closes seam #6 at the boundary):** `decide` writes the
  `knowledge_fingerprint` YOU pass into each delta's `provenance.usage.consumed_knowledge_
  fingerprint`, so the K4 chain closes against the run's real compiled knowledge, not the
  domain-local projection.
- **Trust discipline (seam #1, enforced):** the decisive band is fed only by
  `EvidenceObservation.trusted` (which you set from the certified/QC stream). `decide`
  additionally cross-checks each verdict's `join_id` against `adjudicated_observations`: an
  observation the QC stream **saw and did not mark `TrustLevel.TRUSTED`** is forced
  `trusted=False` → collapses to INSUFFICIENT. The certified stream can only ever REMOVE
  trust, never manufacture it. Set `DiscoveryVerdict.obs_join_id = ObservationObject.obs_id`
  to wire the join; pass `enforce_adjudicated_trust=False` until then (honors your flag).

Tests for this connection point: `tests/test_m28_discovery_v01.py` —
`test_discovery_certification_*` (protocol conformance, decide→deltas mutates nothing,
three-state separation via the kernel gate, knowledge-fingerprint threading, trust-gate
downgrade). All 13 green.
