# M25 generative_construct — seam list (v0.1 skeleton)

Team M25 "Design". Skeleton delivered: `domains/generative_construct/` (objects/provider/
smoke) + `expos/adapters/dry/mutation_operators.py` (deterministic promoter-swap). Smoke
green: `python -m domains.generative_construct` (parent j23103 → 10 promoter-swap children
→ proxy score, deterministic). Reuses `constructs.py` (promoter catalogue) + `sequences.
expression_features` (proxy). No large generative model (later seam).

Seams for the integration owner (B, single-writer of mcl / shared schema):

1. **input_kind** — a generated child is a `sequence_construct` payload (same kind M24
   already routes). No NEW kind needed: the mutation operator emits `{sequence, promoter,
   rbs, cds}`. Confirm the mcl candidate path can accept provider-generated (not just
   preset-catalogue) construct candidates.
2. **candidate-source hook** — mcl's template agent currently proposes from the domain
   `design_space` choices. Seam: let a provider contribute *generated* candidates
   (`GenerativeConstructProvider.propose_children`) into the round's candidate pool, so a
   lineage child can be screened. Needs a `provider.propose_candidates(round, knowledge)`
   entrypoint on the shared provider protocol.
3. **observable** — reuses the existing `expression_proxy` (dry) + `expression_
   fluorescence` (wet) observables; no new observable. The child's proxy is a dry preview.
4. **lineage provenance** — `ConstructDesign.parent_id` / operator id should fold into the
   experiment `DesignProvenance` (generator ≠ `mcl_template_agent`) so a generated
   candidate's ancestry is auditable. Seam: a `generator="generative_construct/v0.1"` +
   parent fingerprint field.
5. **acceptance faces** — v0.1 has the positive proxy-gradient face only. The negative/
   flat face (a swap that does NOT improve proxy) is representable but not yet wired to a
   claim. Later: a "design-move improves expression" claim head.
6. **fingerprints** — the mutation operator is a pure function; its source hash should
   enter `config_fingerprint` when generated candidates are used (like the dry-adapter
   source hash), so a change to the operator flips run identity.
